KFUPM Administrators:
1) Log In: The system shall allow registered administrators to log into the system using valid credentials.
Normal Flow:
a.	The administrator navigates to the Login page.
b.	The administrator clicks on the “Log in as Administrator”
c.	The administrator enters their registered email and password.
d.	The administrator submits the login form.
e.	The system validates the credentials and checks the user role.
f.	If valid, the system authenticates the administrator and grants access to the Administrator page.
2) Log Out: The system shall allow logged-in administrators to log of their account.
Normal Flow:
a.	Administrator selects the profile or account menu.
b.	The administrator selects the “Log Out” option
c.	The system terminates the session.
d.	The system redirects the user to the login/home page.
3) Submit Request to Technical Admin: The system shall provide a “Submit Request” page that allows Administrators to request updates about campus data (routes, buildings, bus schedules, announcements, etc) and to be reviewed/integrated by the Technical Admin team.
Normal Flow:
a.	The administrator logs into the system.
b.	The administrator navigates to the “Request” page.
c.	The administrator selects the request type:
o	Campus Route Add/Update/Delete
o	Campus Building Service Add/Update/Delete
o	Bus Routes/Schedule Update
o	Announcement Add/Edit/Delete
o	Community Path Approval/Removal
o	Hide/delete a comment/rating
o	Discuss/resolve a complaint
o	Others (Please Specify)
d.	The administrator fills in request details based on the chosen request (title, description, affected services, category “for filters”, location, working hours, contacts, optional attachments/links).
e.	The administrator submits the request.
f.	The system validates required fields.
g.	If valid, the system stores the request with status “Pending”.
h.	A confirmation message is displayed, and the admin can view it in “My Requests”.
4) Track Request Status and Communication with Technical Admins: The system shall allow Administrators to track submitted requests and communicate with the Technical Admin team through request comments/updates. Then, make a decision to integrate the update into the system.
Normal Flow:
a.	The administrator logs into the system.
b.	The administrator opens “My Requests.”
c.	The system displays the administrator’s submitted requests with status: Pending / In Progress / Completed / Rejected
d.	The administrator selects a request to view details.
e.	The system displays technical admin responses/updates (if any).
f.	The administrator may add a follow-up comment or clarification (if needed).
g.	The system sends the update made by the administrator and notifies the Technical Admin.
5) Manage Feedback (Interact with Users): The system shall allow KFUPM Administrators to view, respond to, and update the status of user feedback and complaints.
Normal Flow:
a.	The administrator logs into the system.
b.	The administrator navigates to “Feedback & Complaints” page.
c.	The system displays all feedback/complaint entries with filters (date, type, status).
d.	The administrator selects an item to view full details.
e.	The administrator writes a response and assigns a status: Submitted / In Progress / Resolved
f.	The administrator saves changes.
g.	The system updates the record and notifies the user.
6) Respond to Public Reviews (Approve Building Comments): The system shall allow Administrators to oversee and approve or reject user-submitted comments on campus services.
Normal Flow:
a.	The administrator logs into the system.
b.	The administrator navigates to “Public Reviews” Page.
c.	The system displays all building comments.
d.	The administrator selects a comment to review.
e.	The administrator chooses: Approve
f.	The system updates the comment’s status accordingly.
g.	If approved → the comment will have a note ‘Approved by administrators’ visible next to the comment.
h.	If Nothing chosen → the comment remains in public without the admin verification.
7) Receive Submissions from Students about New Paths: The system shall allow Administrators to review student-submitted community paths and forward approved ones to the Technical Admin team.
Normal Flow:
a.	The administrator logs into the system.
b.	The administrator navigates to “Community Paths Review”
c.	The system displays student-submitted paths with status: Approved, Pending, and Rejected.
d.	The administrator selects a path and views:
o	Start & destination
o	Generated route preview (if supported)
o	Title/description
o	Creator info (user)
e.	The administrator chooses:
o	Approve (Send a request to the Technical Team to add it to the system)
o	Reject (not published)
